# گزارش شکاف API — نسخه فارسی ساده

**تاریخ:** ۱۷ ژوئیه ۲۰۲۶  
**محدوده:** بک‌اند `Karzar-main` ↔ فرانت `Storefront` + `admin-panel`

---

## جمع‌بندی یک‌خطی

بک‌اند حدود **۸۷** اندپوینت دارد. فرانت بیشتر مسیرهای روزمرهٔ فروشگاه و ادمین را وصل کرده، ولی چند قابلیت مهم بک‌اند هنوز در UI نیست؛ در عوض چند صفحه/قابلیت فرانت **بدون API واقعی** کار می‌کنند (ماک یا فقط لوکال).

---

## ۱) APIهای بک‌اند که در فرانت پیاده نشده‌اند

### احراز هویت مشتری (Storefront)

| API بک‌اند | معنی ساده | وضعیت فرانت |
|------------|-----------|-------------|
| ثبت‌نام با رمز (`POST /auth/register`) | ساخت حساب با پسورد | نیست — فقط OTP |
| تغییر رمز (`POST /auth/change-password`) | عوض کردن پسورد بعد از ورود | نیست |
| درخواست بازیابی رمز | OTP برای ریست پسورد | نیست |
| تأیید بازیابی رمز | ست کردن پسورد جدید | نیست |

**نتیجه:** مشتری فقط با OTP وارد می‌شود؛ مسیر «رمز عبور / فراموشی رمز» وجود ندارد.

### ادمین — کاربران و امنیت

| API بک‌اند | معنی ساده | وضعیت فرانت |
|------------|-----------|-------------|
| لیست لاگ‌های ممیزی (`GET /users/audit-logs/list`) | تاریخچه اقدامات حساس | نیست |
| حذف نرم کاربر (`DELETE /users/{id}`) | غیرفعال/حذف مشتری | نیست — فقط ویرایش |

### ادمین — انبار و کاتالوگ پیشرفته

| API بک‌اند | معنی ساده | وضعیت فرانت |
|------------|-----------|-------------|
| آمار کاتالوگ (`GET /products/statistics`) | جمع موجودی، ارزش، توزیع | نیست — داشبورد خودش از لیست محصول حساب می‌کند |
| تاریخچه تغییر قیمت/موجودی (`…/change-log`) | لاگ تغییرات محصول | نیست |
| تنظیم موجودی گروهی (`POST …/bulk/stock-adjust`) | چند SKU یکجا | نیست — فقط تک‌محصول |

### ادمین — سفارش

| API بک‌اند | معنی ساده | وضعیت فرانت |
|------------|-----------|-------------|
| آرشیو نرم سفارش (`DELETE /orders/{id}`) | حذف/آرشیو از لیست | نیست |

### عمومی / سیستم (معمولاً لازم نیست در UI)

- `/health`، `/ready`، `/metrics`، ایندکس `/api/v1`
- alias قدیمی بلاگ: `/articles/` (فرانت از `/blog/` استفاده می‌کند — مشکلی نیست)

### چیزهایی که «سرویس دارند» ولی تجربهٔ کامل UI ندارند

- **سبد سرور (`GET/PUT/DELETE /cart`)** در Storefront تعریف شده، ولی منبع اصلی UI همان **Zustand لوکال** است؛ سینک سرور بهترین‌تلاش است، نه منبع حقیقت کامل.
- **تایم‌لاین سفارش** اگر از API نیاید، فرانت خودش تایم‌لاین ساختگی می‌سازد.

---

## ۲) چیزهایی که در فرانت زده‌اید ولی API ندارند (یا فقط ماک/لوکال‌اند)

### Storefront

| قابلیت فرانت | واقعیت |
|--------------|--------|
| صفحه «درباره ما» + آمار و تایم‌لاین شرکت | کاملاً استاتیک — بدون API |
| نوار آمار / ویژگی‌های هوم (FeatureStrip، Why Karzar) | کپی ثابت |
| استعلام سریع پیامکی روی PDP | **موفقیت جعلی** — API صدا زده نمی‌شود |
| قلب علاقه‌مندی روی کارت محصول | فقط ظاهری — بدون wishlist API |
| تنظیمات تماس در فوتر/تماس (شماره، ایمیل، آدرس) | هاردکد — فرم تماس API دارد، اطلاعات تماس نه |
| تصاویر جایگزین picsum | وقتی عکس محصول نیست |
| برچسب‌های مشخصات فنی (fallback) | اگر `spec-labels` قطع شود، لیبل‌های هاردکد |

### Admin

| قابلیت فرانت | واقعیت |
|--------------|--------|
| صفحه **اسناد / Documents** | ۱۰۰٪ ماک — آپلود/دانلود فقط toast |
| صفحه **تنظیمات** | فقط `localStorage` — روی فروشگاه اثر ندارد |
| گزارش‌ها / Reports | از لیست سفارش/محصول در مرورگر جمع می‌شود — API گزارش جدا ندارد |
| داشبورد KPI مالی/سفارش | سفارش‌های واقعی را به درستی نشان نمی‌دهد (از نمونهٔ محصولات می‌سازد) |
| اعتبارنامه‌های تست روی صفحه لاگین | فقط برای توسعه — نباید به پروداکشن برود |

### چیزهایی که «فرانت دارد، بک‌اند هم دارد» ولی ناقص وصل‌اند

| موضوع | توضیح |
|-------|--------|
| مدیریت کامنت‌ها | بک‌اند فقط لیست + حذف دارد؛ فرانت هم همین — **تأیید/رد** در هیچ‌کدام نیست |
| تیکت‌های تماس | بک‌اند فقط لیست؛ فرانت فقط لیست — وضعیت خوانده‌شده/پاسخ‌داده‌شده نیست |
| تنظیمات فروشگاه (نام، تلفن، خاموش‌کردن استعلام) | فقط لوکال ادمین — بک‌اند `/settings` ندارد |

---

## ۳) نقشه پوشش (خلاصه)

| حوزه | پوشش تقریبی |
|------|-------------|
| کاتالوگ خواندن (محصول، دسته، برند، فیلتر) | خوب |
| سبد دو‌لاین + چک‌اوت + پرداخت | خوب (با نکات سینک سبد) |
| OTP فروشگاه + لاگین ادمین + رفرش توکن | خوب |
| سفارش مشتری + ترکینگ + پنل سفارش ادمین | خوب |
| CMS (مقاله، هیرو، کامنت، تماس) | خوب برای CRUD پایه |
| ثبت‌نام/رمز/بازیابی رمز | ضعیف / نیست |
| انبار گروهی + آمار + change-log | نیست |
| اسناد، تنظیمات سراسری، گزارش واقعی | نیست / ماک |

---

## ۴) اولویت پیشنهاد (ساده)

**اول ببندید (شکاف واقعی محصول):**
1. حذف یا وصل کردن «استعلام پیامکی جعلی»
2. مشخص کردن وضعیت Documents و Settings (حذف از منو یا بک‌اند واقعی)
3. مسیر بازیابی رمز / تغییر رمز اگر برای مشتری لازم است
4. pagination کاتالوگ (این بیشتر UI است ولی به `meta` API وابسته است)

**بعداً:**
5. آمار محصول + bulk stock + audit log
6. حذف/آرشیو کاربر و سفارش در ادمین
7. وضعیت تیکت تماس و moderation کامنت (نیاز به گسترش بک‌اند هم دارد)

---

*این گزارش از مقایسهٔ کامل روترهای بک‌اند با سرویس‌های فرانت تهیه شده است.*

# پیوست — پیشنهادهای بهبود از منظر API برای یوزرفلو روان‌تر و UI صادقانه‌تر

> خیلی از مشکلات دیزاین/یوزرفلو فقط با CSS حل نمی‌شوند؛ یا باید API موجود را مصرف کرد، یا قابلیت جعلی را از UI برداشت، یا بک‌اند را گسترش داد. این پیوست همان نقاط را به زبان محصول می‌گوید.

---

## ۱) الان UI دروغ می‌گوید — اول این‌ها را با API یا حذف درست کنید

| UI فعلی | واقعیت API | پیشنهاد |
|---------|------------|---------|
| استعلام پیامکی موفق روی PDP | هیچ اندپوینتی نیست | حذف از UI **یا** API جدید «درخواست تماس/SMS» + وصل |
| Documents آپلود/دانلود | API اسناد نیست | مخفی / «به‌زودی» تا API بیاید |
| Settings نام فروشگاه و خاموش‌کردن استعلام | فقط localStorage | یا Settings سروری، یا برچسب «فقط این دستگاه» |
| امتیاز ثابت ۴.۷ | کامنت‌ها rating دارند | میانگین از `GET …/comments` |
| تایم‌لاین ساختگی گاهی | track/detail می‌تواند timeline بدهد | همیشه اولویت با پاسخ سرور |
| سبد فقط لوکال حس می‌شود | `/cart` موجود است | بعد از لاگین merge + نمایش اختلاف موجودی سرور |

---

## ۲) APIهایی که الان هستند ولی UI برای روان‌سازی باید مصرفشان کند

| API | اثر روی تجربه |
|-----|----------------|
| `GET /products/` + `meta` | صفحه‌بندی / load-more — بدون این کاتالوگ بن‌بست است |
| `GET /products/statistics` | داشبورد ادمین صادق به‌جای جمع ساختگی `base_price` |
| `GET …/stock` + adjust | روی PDP/ادمین موجودی دقیق‌تر و هشدار کمبود |
| `POST /cart/merge` | بعد OTP سبد مهمان گم نشود (الان هست؛ باید در فلو برجسته بماند) |
| `GET /orders/track/{code}` | موفقیت، حساب، و بازیابی پرداخت |
| `POST /payments/init` دوباره | دکمهٔ «پرداخت مجدد» وقتی init اول شکست خورده |
| `GET /categories/spec-labels` | لیبل فارسی فیلترها — fallback هاردکد را کم کنید |

---

## ۳) شکاف‌های بک‌اند که اگر بسته شوند یوزرفلو «کامل» حس می‌شود

| نیاز تجربه | API لازم (الان نیست یا ناقص) |
|------------|-------------------------------|
| فراموشی/تغییر رمز | `password-reset/*`, `change-password` — در بک‌اند هست، در فرانت نه |
| صف اقدامات ادمین | بهتر با statistics + فیلتر سفارش؛ در آینده `/reports` |
| موجودی گروهی انبار | `bulk/stock-adjust` — بک‌اند هست، UI نیست |
| تاریخچه قیمت/موجودی | `change-log` — بک‌اند هست، UI نیست |
| تأیید کامنت قبل از نمایش | moderation — **بک‌اند هم ندارد** |
| وضعیت تیکت تماس | read/replied — **بک‌اند هم ندارد** |
| آدرس ذخیره‌شده مشتری | address book — نیست |
| تنظیمات سراسری سایت | `/settings` یا CMS site config — نیست |
| کتابخانه اسناد | documents API — نیست |

---

## ۴) ترتیب پیشنهادی «API × UX» (بیشترین روان‌سازی)

1. **بدون بک‌اند جدید:** حذف SMS جعلی، Documents، برچسب Settings، امتیاز واقعی، pagination با `meta`، پرداخت مجدد، `?next=`.  
2. **مصرف API موجود ادمین:** statistics، bulk stock، change-log، (اختیاری) حذف کاربر/آرشیو سفارش.  
3. **تصمیم محصول روی auth رمز:** اگر OTP کافی است، مسیر رمز را در docs صریح کنید؛ وگرنه صفحهٔ بازیابی بسازید.  
4. **گسترش مشترک BE+FE:** وضعیت تیکت، moderation کامنت، settings سایت، آدرس‌ها.

---

## ۵) قراردادهای کوچک که دیزاین دو‌لاین را تمیز می‌کند

- همیشه `lane=purchase|inquiry` را در cart و checkout یکسان نگه دارید.  
- برای کالای بدون قیمت در لیست محصول، فیلد/فلگ واضح (نه فقط `price == null` سمت UI).  
- در پاسخ سفارش، `mode` و `payment_status` را در لیست حساب مشتری نشان دهید.  
- Idempotency-Key روی checkout/payment را نگه دارید — برای دکمهٔ «دوباره بزن» حیاتی است.
