# بک زده — فرانت باید مصرف کند (یا هنوز کامل مصرف نکرده)

**تاریخ:** ۱۸ ژوئیه ۲۰۲۶  
**مخاطب:** تیم فرانت / QA  
**هدف:** APIهایی که در بک‌اند `Karzar-main` هستند ولی یا هنوز در UI نیستند، یا فقط ناقص وصل شده‌اند.

> بعد از remediation ژوئیه ۲۰۲۶ خیلی از شکاف‌های قبلی (آمار محصول، bulk stock، change-log، audit log، حذف کاربر، آرشیو سفارش، بازیابی/تغییر رمز) **وصل شدند**. این سند وضعیت **به‌روز** را می‌گوید — نه لیست قدیمی audit.

---

## جمع‌بندی یک‌خطی

بیشتر APIهای روزمره الان در فرانت مصرف می‌شوند. باقی‌مانده‌ها یا **تصمیم محصول**اند (مثل ثبت‌نام با رمز)، یا **عمیق‌سازی** (سبد سرور به‌عنوان منبع حقیقت)، یا **ops** (health/metrics).

---

## ۱) هنوز استفاده نشده / تقریباً استفاده نشده

| API بک‌اند | معنی | وضعیت فرانت الان | باید چه‌کار کنیم |
|------------|------|------------------|------------------|
| `POST /auth/register` | ثبت‌نام با پسورد | نیست (ورود OTP است) | اگر سیاست محصول OTP-only است → در بک غیرفعال بماند و در docs صریح شود. اگر B2B رمز می‌خواهد → صفحه `/register` |
| `GET /articles/` و `GET /articles/{slug}` | alias بلاگ | فرانت از `/blog/` استفاده می‌کند | لازم نیست؛ اختیاری برای سازگاری قدیمی |
| `GET /health` ، `GET /ready` ، `GET /metrics` | سلامت سرویس | در UI محصول نیست | برای دیپلوی/مانیتورینگ (uptime, Prometheus) مصرف شود — نه فروشگاه |
| `GET /` و `GET /api/v1` | ایندکس API | — | فقط دیباگ |

---

## ۲) هست و وصل شده — ولی هنوز باید «عمیق‌تر» مصرف شود

این‌ها دیگر MISSING نیستند؛ کیفیت یکپارچگی را بالا ببرید:

| API | الان | کار باقی‌مانده فرانت |
|-----|------|---------------------|
| `GET/PUT/DELETE /cart` | سرویس هست؛ UI هنوز بیشتر Zustand است | بعد از لاگین اختلاف موجودی سرور را به کاربر نشان دهید؛ قبل چک‌اوت reconcile |
| `GET /orders/track/{code}` + `timeline` | جزئیات حساب از track استفاده می‌کند | همیشه timeline سرور را ترجیح دهید؛ fallback را برچسب «تخمینی» بزنید |
| `GET /products/` + `meta` | load-more کاتالوگ/ادمین | همه لیست‌های ادمین را به pagination یکدست برسانید |
| `GET /categories/spec-labels` | boot فروشگاه | وابستگی به fallback هاردکد را کم کنید |
| `POST /payments/init` | retry بعد از شکست هست | تست E2E با پاک شدن sessionStorage |
| `GET /products/statistics` | داشبورد ادمین | Reports را هم به‌تدریج از آمار سرور تغذیه کنید (تا وقتی `/reports` نیست) |
| `POST …/bulk/stock-adjust` | دیالوگ لیست محصول | UX چندصفحه‌ای + دلیل اجباری |
| `GET …/change-log` | تب ویرایش محصول | فیلتر نوع تغییر (قیمت/موجودی) |
| `GET /users/audit-logs/list` | صفحه audit | لینک از سفارش/کاربر به فیلتر entity |
| `POST /auth/password-reset/*` و `change-password` | `/account/security` | لینک از لاگین فروشگاه به بازیابی؛ پیام خطای واضح‌تر |

---

## ۳) چیزهایی که عمداً مصرف نمی‌کنیم (و درست است)

| API | دلیل |
|-----|------|
| Admin write از Storefront | جداسازی نقش |
| Refund از Storefront | فقط ادمین + step-up |
| `/cms/*` از Storefront | محتوای عمومی از `/blog` و `/hero-slides` |

---

## ۴) چک‌لیست QA سریع

- [ ] با `USE_MOCK=false` آمار داشبورد از `/products/statistics` می‌آید نه جمع ساختگی  
- [ ] bulk stock چند SKU را آپدیت می‌کند  
- [ ] change-log بعد از adjust موجودی ردیف جدید دارد  
- [ ] audit-log بعد از step-up مخرب ردیف دارد  
- [ ] بازیابی رمز و تغییر رمز روی API واقعی خطا/موفقیت درست می‌دهند  
- [ ] سبد مهمان بعد OTP با `cart/merge` قاطی می‌شود  

---

## ۵) اگر بک چیزی را حذف کرد

قبل از deprecate کردن `register` یا alias `/articles`، با فرانت هماهنگ کنید. فرانت فعلی به `/blog` و OTP وابسته است؛ حذف ناگهانی register معمولاً بی‌خطر است اگر `ALLOW_PUBLIC_REGISTER=false` باشد.
